# Jetson NJRH Runtime

This phase adds a temporary field runtime that runs from a repository-owned Jetson workspace while still reusing the validated `car` stack as an upstream asset source.

The control plane now comes from this repository's runtime overlay under `scripts/jetson/runtime_overlay`. The `car` workspace remains an upstream asset source for validated packages, parameters, and helper scripts, but it is no longer the intended owner of runtime orchestration.

## Runtime Goal

- Start a fixed Jetson container named `NJRH-car`
- Keep the runtime inside Docker
- Reuse the existing `web_dashboard` as the temporary operator frontend
- Support live mapping status, 3D map save, 2D map export/view, and navigation startup
- Saving a PGO map from the dashboard now performs a paired save: 3D first, then current `slam_toolbox /map` 2D save plus Isaac localizer asset generation

## Workspace Split

- project-owned Jetson workspace root: `/home/nvidia/workspaces/njrh-v3/workspace1`
- project-owned container mount: `/workspaces/njrh-v3/workspace1`
- upstream asset workspace root: `/home/nvidia/workspaces/isaac_ros-dev`
- upstream asset container mount: `/workspaces/isaac_ros-dev`
- upstream alias mount: `/workspaces/isaac_ros-dev-upstream`

The container now mounts both trees:

- the new repository workspace is the primary runtime root and contains `Dockerfile.car`, `scripts/jetson`, reports, and current package sources
- the historical `isaac_ros-dev` workspace remains mounted read-only as the upstream source of validated `car` scripts, dashboard assets, and installed packages
- the overlay carries `bringup_ranger_can.sh` and `shutdown_ranger_can.sh` compatibility helpers because the reused dashboard calls those filenames directly for CAN up/down operations from inside the container
- structured floor assets are written under `maps_release`. Container startup and common-service startup repair this bind-mounted directory to `admin:admin`, directory mode `2775`, and file mode `664`, so `robot_api_server` can save App-created bundles such as `maps_release/B1/F1/nav` and `maps_release/B1/F1/localizer`.

## Overlay Ownership

- Runtime launcher root: `scripts/jetson/runtime_overlay`
- Dashboard entrypoint owner: `scripts/jetson/runtime_overlay/scripts/run_web_dashboard.sh`
- Dashboard asset staging now uses writable project-owned `maps/`, `maps3d/`, and `waypoints/` directories under `runtime_overlay`, seeded from upstream assets instead of read-only symlinks
- Default Nav2 params owner: `scripts/jetson/runtime_overlay/config/nav2.yaml`
- Jetson CPU affinity owner: `scripts/jetson/runtime_overlay/config/cpu_affinity.env` plus `scripts/jetson/runtime_overlay/scripts/cpu_affinity.sh`. Runtime defaults keep base/safety on CPU1, EKF local odom on CPU2, Nav2 controller/collision on CPU3, the JT128 UDP driver on CPU4, full-density pointcloud remap ingress on CPU5, IMU remap plus localization/local perception on CPU6, `robot_localization_bridge` on CPU7, bursty Nav2 planner/BT work on CPU0, and Nav2 lifecycle supervision on CPU0/CPU1. FAST-LIO2 is not resident during normal navigation. Live 2D mapping re-derives its mode-local CPU sets at launch so FAST-LIO2 frontend/deskew defaults to CPU7 only while mapping is active and PGO/slam_toolbox mapping defaults to CPU7. It also applies a mapping-scoped LiDAR RPS/XPS profile by default (`NJRH_SLAM2D_LIDAR_RPS_XPS_INTERFACE=eth1`, `NJRH_SLAM2D_LIDAR_RPS_XPS_CPUSET=5`) and restores the previous queue masks on exit; API stop/save restores the same state file when it terminates the mapping process group, and this does not write IRQ affinity. Field overrides should use `NJRH_SLAM2D_FASTLIO_CPUSET`, `NJRH_SLAM2D_SLAM_TOOLBOX_CPUSET`, or the `NJRH_SLAM2D_LIDAR_RPS_XPS_*` variables only for deliberate field validation; those cores remain the future elevator-button arm reservation when mapping is not active. `NJRH_CPUSET_AMCL` and `NJRH_CPUSET_AMCL_SCAN_ADMISSION` also default to CPU6; Phase A1.4 starts the C++ `amcl_scan_admission_node` by default with `taskset` and verifies `/proc/<pid>/status` so it does not run on CPU2/CPU3/CPU7 during navigation. Python scan admission is retained only for explicit rollback with `NJRH_AMCL_SCAN_ADMISSION_IMPL=python`.
- Local obstacle runtime owner: `scripts/jetson/runtime_overlay/scripts/run_local_perception.sh`
- Local obstacle runtime is a repo-owned fast filter: in production it runs as a standalone `robot_local_perception` process and consumes `/_internal/lidar_points_local` with best-effort/depth `1` through `NJRH_LOCAL_PERCEPTION_INPUT_PROFILE=local_branch`. `ROBOT_LOCAL_PERCEPTION_INPUT_TOPIC` remains a higher-priority explicit override, and `NJRH_LOCAL_PERCEPTION_INPUT_PROFILE=trunk` returns the same node to canonical `/lidar_points` for rollback/diagnosis. It uses identity input rotation because the JT128 raw-to-canonical axis matrix has already run in `pointcloud_axis_remap_node`, transforms directly into `base_link`, applies range/height/azimuth/self/front masks, then publishes `/perception/obstacle_points` using mode profiles for `NORMAL`, `RAMP`, `ELEVATOR_WAIT`, and `DOORWAY`. The local obstacle branch is latest-only and timer-driven so a slow filter or startup transient cannot process seconds-old clouds or slow the `/lidar_points` publisher.
- Pointcloud acceleration profile owner: `scripts/jetson/runtime_overlay/config/pointcloud_accel_profile.env` plus `scripts/jetson/runtime_overlay/scripts/run_pointcloud_accel_pipeline.sh`. The default is `NJRH_POINTCLOUD_ACCEL_PROFILE=legacy`, which preserves the validated local-branch and nav-scan DDS chain. `ipc_worker` starts the compiled `pointcloud_accel_axis_node`: it publishes the full-density/full-fields `/lidar_points` trunk first, then same-process worker threads publish `/perception/obstacle_points`, `/perception/clearing_points`, and `/scan` from the latest normalized cloud; `laser_scan_to_flatscan` provides `/flatscan`. In accel profiles `/_internal/lidar_points_local` and `/lidar_points_nav` are compact XYZ/XYZI debug/compat topics, and `/points_nav` is not a production hop. `nitros` is a guarded experimental skeleton for compact navigation branches only; if Isaac ROS NITROS is absent, startup exits with a clear recommendation to use `ipc_worker` or `legacy`.
- The C++ local perception node also publishes `/perception/local_perception_status` every two seconds for hardware diagnostics. It reports received/processed/published rates and skip counters without adding a Python probe or changing the obstacle/clearing cloud QoS.
- Pointcloud fan-out diagnosis uses three lightweight C++ status topics: `/lidar/axis_remap_status`, `/perception/local_perception_status`, and `/lidar/nav_cloud_preprocessor_status`. Run `scripts/jetson/runtime_overlay/scripts/set_local_perception_input_profile.sh --profile local_branch --restart` to select the production local branch profile, `--profile trunk --restart` to roll back, `diagnose_lidar_points_jitter.sh` when `/lidar_points` looks low, `inspect_pointcloud_subscribers.sh` to inspect ROS graph fan-out without a PointCloud2 subscription, and `verify_pointcloud_delivery_matrix.sh` for CASE_F/H/I acceptance. The jitter diagnostic defaults to status/graph-only mode; `--include-cli-hz` is the only path that creates a temporary full-density `/lidar_points` subscriber. If status shows low axis publish rate even with the normal branches enabled, run `run_lidar_trunk_pure_ab.sh --execute` only while stationary; it restarts JT128 ingress with both derived branches disabled, collects `/lidar/axis_remap_status`, then restores the production driver profile. Run `check_runtime_process_freshness.sh` after sync/build/restart to catch stale binaries or old running processes. The matrix prefers status topics, only samples `/scan` and `/flatscan` as light topics, and classifies CASE_A/B/C/D/E/F/G/H/I without starting or stopping Nav2. Use `inspect_pointcloud_cpu_affinity.sh` for observation-only PID/TID/core/thermal placement and `record_pointcloud_nav_acceptance.sh --duration-sec 1200` for the 20-minute lightweight report. `CASE_B_LOCAL_DDS_DELIVERY_LOW` means the axis publisher is healthy but the subscriber side is not receiving at target rate; inspect full-density subscribers and run `run_pointcloud_dds_transport_ab.sh` before changing QoS, stamps, or filter geometry.
- `/perception/obstacle_points` and `/perception/clearing_points` are already transformed into `base_link`, and Nav2 consumes both sources as `base_link` clouds. Runtime preserves the input pointcloud acquisition stamp (`restamp_to_now=false`, `restamp_to_latest_tf=false`, `require_output_stamp_tf=false`) so any JT128/DDS/perception queue delay remains visible as topic age instead of being hidden by current-stamped obstacle clouds. The local costmap uses `global_frame=base_link`, avoiding a per-scan historical `odom <- base_link` lookup inside the Humble VoxelLayer MessageFilter while leaving global planning, localization, and the canonical TF tree unchanged.
- Resident navigation startup no longer runs a blocking TF/topic probe before Nav2 launch. Local perception stays resident, obstacle clouds keep their source stamps, and any TF/costmap timing fault is diagnosed from the running system instead of preventing Nav2 from starting.
- The fresh `odom -> base_link` gate also watches the live `/tf` stream for that exact dynamic transform and only uses tf2 lookup as a fallback. This keeps the strict 0.25 s freshness contract while avoiding false stale reports from a newly-created tf2 buffer during ROS discovery.
- Navigation helper reuse stays process-first for non-TF helpers, but `robot_local_state` is treated as a canonical TF owner and must also expose its ROS endpoint, `/local_state/odometry`, and a fresh `odom -> base_link` TF before reuse. Detailed local perception timing and QoS validation remains available through explicit diagnostics, but it is not a pre-launch gate.
- `robot_local_state` republishes the latest valid EKF odom sample at 30 Hz for up to 0.50 s (`republish_latest=true`). This gives Nav2 a dense `odom -> base_link` TF cache while still surfacing real odom outages after the stale timeout.
- Local costmap clearing uses separated sources: `/perception/obstacle_points` marks obstacles only, while `/perception/clearing_points` publishes decimated synthetic ray endpoints from the live `lidar_link` origin in `base_link`, so moved obstacles can clear even when the next JT128 scan has no real return behind them. The synthetic clearing fan includes near and far range steps (`0.50/1.00/2.00/3.50/5.50/8.00 m`) across dense low z layers; it is published at a lower cadence than obstacle marking so clearing raytrace work cannot delay the safety-critical obstacle cloud.
- The Nav2 local `VoxelLayer` keeps `z_resolution=0.10` and now uses `mark_threshold=2`, so a single marked voxel column is no longer enough to create a 2D occupied cell. This specifically reduces the small residual spots that remain after clearing when only one low voxel is left marked.
- Local-costmap-only debug mode is owned by `scripts/jetson/runtime_overlay/scripts/run_local_costmap_debug.sh` plus `src/robot_bringup/launch/local_costmap_debug.launch.py`. The dashboard button `只启动局部障碍地图` starts the JT128/chassis/TF prerequisites, `robot_local_perception`, and only Nav2 `controller_server` with `/local_costmap/costmap`; it intentionally does not start planner, BT navigator, velocity smoother, collision monitor, robot safety, or final `/cmd_vel`.
- Final command arbitration runtime owner: `scripts/jetson/runtime_overlay/scripts/run_robot_safety.sh`
- Final command arbitration runtime now exposes `/safety/status` and `/safety/motion_allowed` and publishes safe commands to `/cmd_vel_safe`, so field runtime can distinguish `ESTOP_ACTIVE`, `LOCALIZATION_INVALID`, and `COMMAND_STALE`
- Local odometry runtime owner: `scripts/jetson/runtime_overlay/scripts/run_local_state.sh`. Field runtime now defaults to `LOCAL_STATE_MODE=ekf`: Ranger `/wheel/odom` is normalized to `/wheel/odom_ekf`, `/lidar_imu` is bias-corrected to `/lidar_imu_bias_corrected`, and `robot_localization` fuses wheel x/y/yaw pose, wheel forward/yaw velocity, and JT128 IMU yaw-rate into `/local_state/odometry` while owning the only `odom -> base_link` TF. For temporary chassis-odom isolation, `LOCAL_STATE_EKF_PROFILE=wheel_only` keeps the same EKF/TF owner but selects `local_state_ekf_wheel_only.yaml`, skips the IMU gyro-bias filter, and removes EKF `imu0` fusion so `/local_state/odometry` comes from `/wheel/odom_ekf` only. The Jetson override is persisted in `scripts/jetson/runtime_overlay/config/local_state_ekf_profile.env`; restore the default by changing it to `wheel_imu`. FAST-LIO local-state remains an explicit diagnostic mode with `LOCAL_STATE_MODE=fastlio`; it requires an explicitly managed FAST-LIO runtime, converts `/Odometry` through `robot_fastlio_mapping/fastlio_odom_bridge_node` into `/fastlio/base_odometry`, and then lets `robot_local_state` republish it. Canonical helper wrappers disown a launched helper and remove its PID from wrapper-owned cleanup, so a failed Nav2/localizer mode wrapper cannot SIGHUP-kill `run_local_state.sh` or leave `/tf` without the `odom -> base_link` publisher.
- Runtime DDS defaults are centralized in `scripts/jetson/runtime_overlay/scripts/common_env.sh`: overlay helpers use `rmw_fastrtps_cpp` with `FASTDDS_BUILTIN_TRANSPORTS=UDPv4`. This keeps ROS discovery on UDPv4 and avoids Fast DDS shared-memory lock failures where a process remains alive but its node and publishers never appear in the graph. `common_env.sh` also writes a default Fast DDS XML profile and exports both `FASTRTPS_DEFAULT_PROFILES_FILE` and `FASTDDS_DEFAULT_PROFILES_FILE`; the default DDS interface whitelist is `lo,wlan0`, so `eth1`, `docker0`, USB, and link-local interfaces do not join ROS discovery unless `NJRH_FASTDDS_ALLOWED_INTERFACES` is explicitly changed. If the default `/tmp/njrh_fastdds_profile.xml` is owned by a different runtime user and is not writable, the helper falls back to `/tmp/njrh_fastdds_profile_<uid>.xml` instead of failing startup. Set `NJRH_FASTDDS_PROFILE_ENABLED=false` only for diagnostics. `common_env.sh` sets `NJRH_COMMON_ENV_SETUP_DONE=1` after the first ROS/upstream/patched-overlay/project-install setup, so nested runtime helpers inherit the prepared environment and do not block navigation startup by repeatedly running `_local_setup_util.py`; direct project binaries such as `robot_floor_manager` also get their generated `robot_interfaces` typesupport libraries on `LD_LIBRARY_PATH`.
- Runtime health snapshot debug tool: `scripts/jetson/runtime_overlay/scripts/runtime_health_guard.py`. Common services do not start it by default; enable it only with `NJRH_RUNTIME_HEALTH_GUARD_AUTOSTART=true` during field debugging. The compiled `robot_bringup/runtime_readiness_probe` one-shot binary remains available for explicit diagnostics and for the bounded navigation startup sequence gate, but normal navigation startup does not repeatedly create short-lived DDS participants. When explicitly enabled, the guard observes lightweight `/local_state/odometry`, one `/tf` stream, `/localization_result`, and `/safety/status`; `/fastlio/base_odometry` is relevant only in explicit FAST-LIO diagnostic local-state mode.
- Occupancy localization no longer runs a periodic `robot_localization_bridge` ROS-graph watchdog. That watchdog created short-lived Python/rclpy probes during navigation runtime and could compete with the TF/localization chain. Startup sends a bounded localization trigger request, then process supervision and API goal admission own recovery instead of a polling Python loop.
- Isaac localization has two timing policies. Service-triggered relocalization through `/global_localization/trigger` can accept a `/localization_result` whose header stamp is several seconds older than receive time because Isaac reports the original scan/localizer stamp; the bridge still uses that stamp for historical `odom -> base_link` lookup and does not restamp. Continuous localization modes use the stricter `continuous_max_result_age_ms` gate before any gated correction can affect `map -> odom`. The trigger wrapper now waits for bridge acceptance, `has_map_to_odom=true`, and a live `map -> odom` from `robot_localization_bridge`, not only for the direct Isaac service call to return.
- AMCL continuous correction is controlled separately by `NJRH_AMCL_LOCALIZATION_MODE=disabled|shadow|gated` and defaults to `gated` for production correction; `shadow` remains the explicit rollback/diagnostic mode that reports candidates without changing `map -> odom`. The AMCL profile uses `/scan_amcl`, `map_topic=/map`, and `tf_broadcast=false`; `/flatscan` remains Isaac-triggered-only. `/scan_amcl` is an AMCL production admission input, not a debug topic: it relays `/scan` without restamping or changing frame/ranges/intensities/scan metadata, drops stale scans and scans without `odom <- scan_frame` at the original stamp, defaults to 5 Hz, and is pinned by `NJRH_CPUSET_AMCL_SCAN_ADMISSION` to the localization CPU set. The default implementation is C++ (`NJRH_AMCL_SCAN_ADMISSION_IMPL=cpp`, executable `robot_localization_bridge/amcl_scan_admission_node`); Python `amcl_scan_admission_relay.py` is an explicit temporary rollback and a missing C++ binary fails startup instead of silently falling back. During navigation startup, AMCL now starts resident with the localization layer before the initial Isaac trigger; after the trigger is accepted and `map -> odom` is live, the runtime seeds AMCL from the bridge-approved `map -> base_link` and calls `/request_nomotion_update` if the robot is stationary. The no-motion readiness helper `amcl_nomotion_update_probe.py` subscribes to `/amcl_pose` before the service call so it cannot miss AMCL's single response, and `/tmp/njrh_amcl_runtime_status.env` has a TTL so stale startup `AMCL_FAILED` does not override live AMCL evidence. AMCL static stale pose is normal static standby; moving stale pose is reported as AMCL not tracking, and gated correction is enabled only when `amcl_correction_ready=true`. Inspect live placement with `bash scripts/jetson/runtime_overlay/scripts/inspect_runtime_cpu_affinity.sh`; check readiness with `bash scripts/jetson/runtime_overlay/scripts/verify_amcl_nomotion_readiness.sh --mode gated --expect-static-standby`; observe a moving navigation window with `bash scripts/jetson/runtime_overlay/scripts/observe_amcl_navigation_shadow_180s.sh --duration-sec 180`. Roll back scan admission with `export NJRH_AMCL_SCAN_ADMISSION_IMPL=python` plus `bash scripts/jetson/runtime_overlay/scripts/run_amcl_shadow_localization.sh --restart`; disable correction but keep AMCL observation with `export NJRH_AMCL_LOCALIZATION_MODE=shadow` and restart the navigation runtime; disable AMCL with `export NJRH_AMCL_LOCALIZATION_MODE=disabled` and `bash scripts/jetson/runtime_overlay/scripts/run_amcl_shadow_localization.sh --stop`.
- Phase L2 post-relocalization settle is owned by `robot_api_server` and enabled by `post_relocalization_settle_enabled=true` in `scripts/jetson/runtime_overlay/config/robot_api_server.yaml`. After explicit forced Isaac relocalization is accepted, it waits on bridge sequence/owner/freshness, `odom -> base_link`, `base_link -> lidar_level_link`, local-costmap heartbeat, and local-costmap MessageFilter stability before sending post-undock/predock Nav2 goals or entering fine docking. Phase TF1 adds a bridge-publisher heartbeat to that gate: `robot_localization_bridge` correction callbacks only update `MapOdomState`, while the independent 50 Hz publisher callback group is the only `sendTransform()` path. The barrier now also requires `publisher_decoupled_from_correction=true`, a valid state, accepted/published sequence alignment, and `map_odom_publish_gap_ms <= 250`. Verify read-only with `bash scripts/jetson/runtime_overlay/scripts/verify_post_relocalization_settle_barrier.sh --reason manual --duration-sec 30 --watch` and `bash scripts/jetson/runtime_overlay/scripts/verify_bridge_map_odom_publisher.sh --duration-sec 10`; observe a live flow with `bash scripts/jetson/runtime_overlay/scripts/observe_relocalization_to_next_stage.sh --duration-sec 120` or `bash scripts/jetson/runtime_overlay/scripts/observe_tf_stability_after_relocalization.sh --duration-sec 180 --label <run>`. Roll back by setting `post_relocalization_settle_enabled: false` for the API gate, or by reverting the TF1 code/config patch; this does not change Nav2 controller/planner, TF tolerances, pointcloud QoS/DDS, FAST-LIO2, Ranger odom, or EKF.
- Phase U1 applies that same design specifically to auto-undock before normal point navigation. After `/docking/undock` succeeds and post-undock relocalization is accepted, the original pending Nav2 goal remains held until `post_undock_relocalization_settle_enabled=true` passes the post-undock barrier. This is necessary because API-side TF readiness and bridge acceptance do not guarantee that `controller_server` and the hosted local costmap have a warmed TF buffer. `/api/v1/navigation/state.post_undock_settle` and `/api/v1/docking/state.docking` expose whether the pending goal is held or released, the settle failure reason, AMCL readiness, and localization degradation. Observe a live docked-to-navigation request with `bash scripts/jetson/runtime_overlay/scripts/observe_post_undock_to_nav_goal.sh --duration-sec 180`; the script is read-only, does not send goals, and does not subscribe to heavy point clouds. Roll back by setting `post_undock_relocalization_settle_enabled: false` and restarting `robot_api_server`.
- `scripts/jetson/njrh_container.sh stop-common` now waits for `run_common_services.sh` to exit after SIGINT, then escalates to SIGTERM and SIGKILL if needed. A stale common wrapper should no longer make the next `start-common` reuse an old failed runtime state.

- Ranger Mini 3 mode adaptation runtime owner: `scripts/jetson/runtime_overlay/scripts/run_ranger_mini3_mode_controller.sh`. - App API runtime owner: `scripts/jetson/runtime_overlay/scripts/run_robot_api_server_supervised.sh`. - The API server keeps its `/tf` subscription resident at startup. `/api/v1/robot/pose` polling and App page subscription release must not repeatedly create or destroy reliable `/tf` subscribers, because that Fast DDS endpoint churn can backpressure `robot_localization_bridge` while the bridge PID still exists.

- Repository-owned bringup contract now exists in `src/robot_bringup/launch/localization_bringup.launch.py` and `navigation_bringup.launch.py`; Jetson shell helpers still wrap upstream scripts today, but they now have a repo launch target to converge to

## Frontend Flow

After opening the dashboard, the current field-default mapping flow is the existing 2D path:

1. Start 2D mapping to launch the repository-owned `slam_toolbox` chain.
2. Open the live 2D map view to inspect `/map`.
3. Use the standalone 2D save path to save the live `slam_toolbox` map and generate the matching Isaac localizer png/yaml beside it.
4. Open the saved 2D map view to inspect the navigation asset.
5. Select a 2D map and start standard navigation.

The 3D/PGO controls remain available for optional formal mapping: `开始3D建图` still launches JT128 + FAST-LIO2 + PGO + live `slam_toolbox`, and the paired 3D save still snapshots the current `slam_toolbox /map` after the PGO 3D asset is saved. This is not the current default field workflow.

For obstacle-layer debugging only, click `只启动局部障碍地图`. This opens the local costmap view (`source=local_costmap`) and should be used before full navigation when verifying `/perception/obstacle_points` marking and `/perception/clearing_points` clearing behavior.

Dynamic avoidance uses a two-stage policy. `/perception/obstacle_points` marks the local voxel costmap, while `/perception/clearing_points` clears stale voxels. Standard navigation wraps MPPI with `RotationShimController`: only large heading errors (`1.20 rad`) and final yaw alignment are handled by direct heading alignment / turn-in-place behavior, while MPPI remains the primary controller for path following and obstacle avoidance. MPPI is still configured with an Ackermann motion model. The downstream C++ Ranger mode controller forbids crab/lateral output and reverse during navigation, keeps moving high-curvature requests in Ackermann mode with yaw-rate clamping, and allows automatic spin only at very low speed or for pure-yaw commands. Crab mode is reserved for `/docking/start` near-field charger alignment. `collision_monitor` should only hard-stop for close body-risk returns and should otherwise slow the robot rather than suppress forward motion.

The live MPPI envelope is constrained to the Ranger Mini 3 documented Ackermann radius and the current field speed target: `AckermannConstraints.min_turning_r=0.81`, `vx_max=1.20`, `wz_max=0.70`, `vx_std=0.45`, `wz_std=0.45`. The local planning window is `10m x 10m`, the MPPI horizon is `44 * 0.0833333333s = 3.67s`, matching the 12 Hz controller period, `ObstaclesCritic` is matched to `local_inflation_layer` (`inflation_radius=0.35`, `cost_scaling_factor=6.0`), and `PathAlignCritic.max_path_occupancy_ratio=0.05` so local dynamic obstacle avoidance can deviate from the global path earlier. Single-pose navigation uses the repository stable behavior tree to avoid periodic path replacement resetting MPPI on an unchanged goal.

The legacy `robot_local_perception` balanced clearing profile is `clearing.virtual_rays.angular_resolution_deg=0.75`, `range_steps=[0.50, 1.00, 2.00, 3.50, 5.50, 8.00]`, 8 z endpoints, and `clearing.max_points=30000`. In accel profiles, the same clearing role is owned by `PointCloudAccelCore` through `clearing_worker_virtual_rays_enabled=true`, `clearing_worker_virtual_ray_angle_resolution_deg=1.0`, the same range steps, 8 z endpoints, and `clearing_worker_max_points=15000`.
