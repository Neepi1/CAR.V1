# Jetson NJRH Runtime

This phase adds a temporary field runtime that runs from a repository-owned Jetson workspace while still reusing the validated `car` stack as an upstream asset source.

The control plane now comes from this repository's runtime overlay under `scripts/jetson/runtime_overlay`. The `car` workspace remains an upstream asset source for validated packages, parameters, and helper scripts, but it is no longer the intended owner of runtime orchestration.

## Runtime Goal

- Start a fixed Jetson container named `NJRH-car`
- Keep the runtime inside Docker
- Reuse the existing `web_dashboard` as the temporary operator frontend
- Support live mapping status, 3D map save, 2D map export/view, and navigation startup
- Saving a PGO map from the dashboard now performs a paired save: 3D first, then current `slam_toolbox /map` 2D save plus Isaac localizer asset generation

The container now mounts both trees:

- the new repository workspace is the primary runtime root and contains `Dockerfile.car`, `scripts/jetson`, reports, and current package sources
- the historical `isaac_ros-dev` workspace remains mounted read-only as the upstream source of validated `car` scripts, dashboard assets, and installed packages
- the overlay carries `bringup_ranger_can.sh` and `shutdown_ranger_can.sh` compatibility helpers because the reused dashboard calls those filenames directly for CAN up/down operations from inside the container
- structured floor assets are written under `maps_release`. Container startup and common-service startup repair this bind-mounted directory to `admin:admin`, directory mode `2775`, and file mode `664`, so `robot_api_server` can save App-created bundles such as `maps_release/B1/F1/nav` and `maps_release/B1/F1/localizer`.

- Jetson CPU affinity owner: `scripts/jetson/runtime_overlay/config/cpu_affinity.env` plus `scripts/jetson/runtime_overlay/scripts/cpu_affinity.sh`. Runtime defaults keep base/safety on CPU1, EKF local odom on CPU2, Nav2 controller/collision on CPU3, the JT128 UDP driver on CPU4, full-density pointcloud remap ingress on CPU5, IMU remap plus localization/local perception on CPU6, `robot_localization_bridge` on CPU7, bursty Nav2 planner/BT work on CPU0, and Nav2 lifecycle supervision on CPU0/CPU1. FAST-LIO2 is not resident during normal navigation. Live 2D mapping re-derives its mode-local CPU sets at launch so FAST-LIO2 frontend/deskew defaults to CPU7 only while mapping is active and PGO/slam_toolbox mapping defaults to CPU7. It also applies a mapping-scoped LiDAR RPS/XPS profile by default (`NJRH_SLAM2D_LIDAR_RPS_XPS_INTERFACE=eth1`, `NJRH_SLAM2D_LIDAR_RPS_XPS_CPUSET=5`) and restores the previous queue masks on exit; API stop/save restores the same state file when it terminates the mapping process group, and this does not write IRQ affinity. Field overrides should use `NJRH_SLAM2D_FASTLIO_CPUSET`, `NJRH_SLAM2D_SLAM_TOOLBOX_CPUSET`, or the `NJRH_SLAM2D_LIDAR_RPS_XPS_*` variables only for deliberate field validation; those cores remain the future elevator-button arm reservation when mapping is not active. `NJRH_CPUSET_AMCL` and `NJRH_CPUSET_AMCL_SCAN_ADMISSION` also default to CPU6; Phase A1.4 starts the C++ `amcl_scan_admission_node` by default with `taskset` and verifies `/proc/<pid>/status` so it does not run on CPU2/CPU3/CPU7 during navigation. Python scan admission is retained only for explicit rollback with `NJRH_AMCL_SCAN_ADMISSION_IMPL=python`.
- Local obstacle runtime owner: `scripts/jetson/runtime_overlay/scripts/run_local_perception.sh`
- Local obstacle runtime is a repo-owned fast filter: in production it runs as a standalone `robot_local_perception` process and consumes `/_internal/lidar_points_local` with best-effort/depth `1` through `NJRH_LOCAL_PERCEPTION_INPUT_PROFILE=local_branch`. `ROBOT_LOCAL_PERCEPTION_INPUT_TOPIC` remains a higher-priority explicit override, and `NJRH_LOCAL_PERCEPTION_INPUT_PROFILE=trunk` returns the same node to canonical `/lidar_points` for rollback/diagnosis. It uses identity input rotation because the JT128 raw-to-canonical axis matrix has already run in `pointcloud_axis_remap_node`, transforms directly into `base_link`, applies range/height/azimuth/self/front masks, then publishes `/perception/obstacle_points` using mode profiles for `NORMAL`, `RAMP`, `ELEVATOR_WAIT`, and `DOORWAY`. The local obstacle branch is latest-only and timer-driven so a slow filter or startup transient cannot process seconds-old clouds or slow the `/lidar_points` publisher.
- Pointcloud acceleration profile owner: `scripts/jetson/runtime_overlay/config/pointcloud_accel_profile.env` plus `scripts/jetson/runtime_overlay/scripts/run_pointcloud_accel_pipeline.sh`. The default is `NJRH_POINTCLOUD_ACCEL_PROFILE=legacy`, which preserves the validated local-branch and nav-scan DDS chain. `ipc_worker` starts the compiled `pointcloud_accel_axis_node`: it publishes the full-density/full-fields `/lidar_points` trunk first, then same-process worker threads publish `/perception/obstacle_points`, `/perception/clearing_points`, and `/scan` from the latest normalized cloud; `laser_scan_to_flatscan` provides `/flatscan`. In accel profiles `/_internal/lidar_points_local` and `/lidar_points_nav` are compact XYZ/XYZI debug/compat topics, and `/points_nav` is not a production hop. `nitros` is a guarded experimental skeleton for compact navigation branches only; if Isaac ROS NITROS is absent, startup exits with a clear recommendation to use `ipc_worker` or `legacy`.
- The C++ local perception node also publishes `/perception/local_perception_status` every two seconds for hardware diagnostics. It reports received/processed/published rates and skip counters without adding a Python probe or changing the obstacle/clearing cloud QoS.
- Pointcloud fan-out diagnosis uses three lightweight C++ status topics: `/lidar/axis_remap_status`, `/perception/local_perception_status`, and `/lidar/nav_cloud_preprocessor_status`. Run `scripts/jetson/runtime_overlay/scripts/set_local_perception_input_profile.sh --profile local_branch --restart` to select the production local branch profile, `--profile trunk --restart` to roll back, `diagnose_lidar_points_jitter.sh` when `/lidar_points` looks low, `inspect_pointcloud_subscribers.sh` to inspect ROS graph fan-out without a PointCloud2 subscription, and `verify_pointcloud_delivery_matrix.sh` for CASE_F/H/I acceptance. The jitter diagnostic defaults to status/graph-only mode; `--include-cli-hz` is the only path that creates a temporary full-density `/lidar_points` subscriber. If status shows low axis publish rate even with the normal branches enabled, run `run_lidar_trunk_pure_ab.sh --execute` only while stationary; it restarts JT128 ingress with both derived branches disabled, collects `/lidar/axis_remap_status`, then restores the production driver profile. Run `check_runtime_process_freshness.sh` after sync/build/restart to catch stale binaries or old running processes. The matrix prefers status topics, only samples `/scan` and `/flatscan` as light topics, and classifies CASE_A/B/C/D/E/F/G/H/I without starting or stopping Nav2. Use `inspect_pointcloud_cpu_affinity.sh` for observation-only PID/TID/core/thermal placement and `record_pointcloud_nav_acceptance.sh --duration-sec 1200` for the 20-minute lightweight report. `CASE_B_LOCAL_DDS_DELIVERY_LOW` means the axis publisher is healthy but the subscriber side is not receiving at target rate; inspect full-density subscribers and run `run_pointcloud_dds_transport_ab.sh` before changing QoS, stamps, or filter geometry.
- `/perception/obstacle_points` and `/perception/clearing_points` are already transformed into `base_link`, and Nav2 consumes both sources as `base_link` clouds. Runtime publish-time stamps these derived local-costmap outputs so Nav2's `global_frame=odom` local costmap does not need to transform current `base_link` geometry at an older source acquisition stamp. Source stamp age remains visible in the local perception / pointcloud accel status fields for JT128/DDS/perception queue diagnostics.
- Resident navigation startup no longer runs a blocking TF/topic probe before Nav2 launch. Local perception stays resident, local-costmap clouds carry publish-time stamps, and any TF/costmap timing fault is diagnosed from the running system instead of preventing Nav2 from starting.
- The fresh `odom -> base_link` gate also watches the live `/tf` stream for that exact dynamic transform and only uses tf2 lookup as a fallback. This keeps the strict 0.25 s freshness contract while avoiding false stale reports from a newly-created tf2 buffer during ROS discovery.
- Navigation helper reuse stays process-first for non-TF helpers, but `robot_local_state` is treated as a canonical TF owner and must also expose its ROS endpoint, `/local_state/odometry`, and a fresh `odom -> base_link` TF before reuse. Detailed local perception timing and QoS validation remains available through explicit diagnostics, but it is not a pre-launch gate.
- `robot_local_state` republishes the latest valid EKF odom sample at 30 Hz for up to 0.50 s (`republish_latest=true`). This gives Nav2 a dense `odom -> base_link` TF cache while still surfacing real odom outages after the stale timeout.
- Local costmap clearing uses separated sources: `/perception/obstacle_points` marks obstacles only, while `/perception/clearing_points` publishes decimated synthetic ray endpoints from the live `lidar_link` origin in `base_link`, so moved obstacles can clear even when the next JT128 scan has no real return behind them. The synthetic clearing fan uses the same angular observation window as obstacle marking, starts at `0.10m`, includes near and far range steps up to `8.00m`, and uses multiple z endpoints; it is published at a lower cadence than obstacle marking so clearing raytrace work cannot delay the safety-critical obstacle cloud. The local VoxelLayer explicitly enables footprint clearing so stale cells inside the robot footprint are not retained.
- The Nav2 local `VoxelLayer` keeps `z_resolution=0.10` and now uses `mark_threshold=2`, so a single marked voxel column is no longer enough to create a 2D occupied cell. This specifically reduces the small residual spots that remain after clearing when only one low voxel is left marked.

Field runtime currently defaults to `LOCAL_STATE_MODE=ekf LOCAL_STATE_EKF_PROFILE=wheel_only` as the Ranger odom safety baseline: Ranger `/wheel/odom` is normalized to `/wheel/odom_ekf`, and `robot_localization` republishes it as `/local_state/odometry` while `robot_local_state` owns the only `odom -> base_link` TF without IMU fusion. - Runtime DDS defaults are centralized in `scripts/jetson/runtime_overlay/scripts/common_env.sh`: overlay helpers use `rmw_fastrtps_cpp` with `FASTDDS_BUILTIN_TRANSPORTS=UDPv4`. This keeps ROS discovery on UDPv4 and avoids Fast DDS shared-memory lock failures where a process remains alive but its node and publishers never appear in the graph. `common_env.sh` also writes a default Fast DDS XML profile and exports both `FASTRTPS_DEFAULT_PROFILES_FILE` and `FASTDDS_DEFAULT_PROFILES_FILE`; the default DDS interface whitelist is `lo,wlan0`, so `eth1`, `docker0`, USB, and link-local interfaces do not join ROS discovery unless `NJRH_FASTDDS_ALLOWED_INTERFACES` is explicitly changed. If the default `/tmp/njrh_fastdds_profile.xml` is owned by a different runtime user and is not writable, the helper falls back to `/tmp/njrh_fastdds_profile_<uid>.xml` instead of failing startup. - Runtime health snapshot debug tool: `scripts/jetson/runtime_overlay/scripts/runtime_health_guard.py`. - Occupancy localization no longer runs a periodic `robot_localization_bridge` ROS-graph watchdog. That watchdog created short-lived Python/rclpy probes during navigation runtime and could compete with the TF/localization chain. Startup sends a bounded localization trigger request, then process supervision and API goal admission own recovery instead of a polling Python loop.
- Isaac localization has two timing policies. Service-triggered relocalization through `/global_localization/trigger` can accept a `/localization_result` whose header stamp is several seconds older than receive time because Isaac reports the original scan/localizer stamp; the bridge still uses that stamp for historical `odom -> base_link` lookup and does not restamp. Continuous localization modes use the stricter `continuous_max_result_age_ms` gate before any gated correction can affect `map -> odom`. The trigger wrapper now waits for bridge acceptance, `has_map_to_odom=true`, and a live `map -> odom` from `robot_localization_bridge`, not only for the direct Isaac service call to return.
- AMCL continuous correction is controlled separately by `NJRH_AMCL_LOCALIZATION_MODE=disabled|shadow|gated` and defaults to `gated` for commercial navigation: `NJRH_AMCL_ACCEPT_CORRECTIONS_WHILE_MOVING=true` lets bounded AMCL small/medium corrections update `map -> odom` during normal navigation. Direct AMCL corrections are limited to `0.07m`, medium corrections are limited to `0.30m` after 3 consistent candidates, and larger AMCL offsets are left for explicit Isaac recovery instead of being applied as continuous drift correction. Use `NJRH_AMCL_LOCALIZATION_MODE=shadow` only for odom/navigation audits where AMCL remains resident and produces candidates without updating `map -> odom`. The AMCL profile uses `/scan_amcl`, `map_topic=/map`, and `tf_broadcast=false`; `/flatscan` remains Isaac-triggered-only. `/scan_amcl` is an AMCL production admission input, not a debug topic: it relays `/scan` without restamping or changing frame/ranges/intensities/scan metadata, drops stale scans and scans without `odom <- scan_frame` at the original stamp, defaults to 5 Hz, and is pinned by `NJRH_CPUSET_AMCL_SCAN_ADMISSION` to the localization CPU set. The default implementation is C++ (`NJRH_AMCL_SCAN_ADMISSION_IMPL=cpp`, executable `robot_localization_bridge/amcl_scan_admission_node`); Python `amcl_scan_admission_relay.py` is an explicit temporary rollback and a missing C++ binary fails startup instead of silently falling back. During navigation startup, AMCL now starts resident with the localization layer before the initial Isaac trigger; after the trigger is accepted and `map -> odom` is live, the runtime seeds AMCL from the bridge-approved `map -> base_link` and calls `/request_nomotion_update` if the robot is stationary. The no-motion readiness helper `amcl_nomotion_update_probe.py` subscribes to `/amcl_pose` before the service call so it cannot miss AMCL's single response, and `/tmp/njrh_amcl_runtime_status.env` has a TTL so stale startup `AMCL_FAILED` does not override live AMCL evidence. AMCL static stale pose is normal static standby; moving stale pose is reported as AMCL not tracking. In gated mode, seeded/static standby can report `amcl_ready=true` with `amcl_correction_pending=true` until a fresh correction exists; actual correction application still requires `amcl_correction_ready=true`. Inspect live placement with `bash scripts/jetson/runtime_overlay/scripts/inspect_runtime_cpu_affinity.sh`; check readiness with `bash scripts/jetson/runtime_overlay/scripts/verify_amcl_nomotion_readiness.sh --mode gated --expect-static-standby`; observe a moving navigation window with `bash scripts/jetson/runtime_overlay/scripts/observe_amcl_navigation_shadow_180s.sh --duration-sec 180`. Roll back scan admission with `export NJRH_AMCL_SCAN_ADMISSION_IMPL=python` plus a full `sudo systemctl restart njrh-runtime.service`; audit odom-only with `export NJRH_AMCL_LOCALIZATION_MODE=shadow` and restart the navigation runtime; disable AMCL with `export NJRH_AMCL_LOCALIZATION_MODE=disabled` and restart the navigation runtime.

- Phase L2 post-relocalization settle is owned by `robot_api_server` and enabled by `post_relocalization_settle_enabled=true` in `scripts/jetson/runtime_overlay/config/robot_api_server.yaml`. - Phase U1 applies that same design specifically to auto-undock before normal point navigation. Dynamic avoidance uses a two-stage policy. `/perception/obstacle_points` marks the local voxel costmap, while `/perception/clearing_points` clears stale voxels. Standard navigation wraps MPPI with `RotationShimController`: path-entry heading errors above `0.45 rad` and final yaw alignment are handled by direct heading alignment / turn-in-place behavior, while MPPI remains the primary controller for path following and obstacle avoidance. MPPI is still configured with an Ackermann motion model, and RotationShim disengages below `0.075 rad` so ordinary mid-route turns do not enter shim too early, but a real shim turns close enough before handing back to MPPI. The shim pure-yaw cap is `0.60 rad/s`, matching the API final-yaw cap used after near-goal handoff; residual spin tail is handled downstream by `robot_safety` using actual `/wheel/odom` yaw-rate settle. The downstream C++ Ranger mode controller forbids crab/lateral output and reverse during navigation, keeps moving high-curvature requests in Ackermann mode with yaw-rate clamping, and allows automatic spin only at very low speed or for pure-yaw commands. Crab mode is reserved for `/docking/start` near-field charger alignment. `collision_monitor` should only hard-stop for close body-risk returns and should otherwise slow the robot rather than suppress forward motion.

The live MPPI envelope is constrained to the Ranger Mini 3 documented Ackermann radius and the current field speed target: `AckermannConstraints.min_turning_r=0.81`, `vx_max=1.20`, and `wz_max=0.70`. To match the measured chassis response, MPPI uses a 4.0 s horizon (`time_steps=48`, `model_dt=0.0833333333`) and lower command sampling noise (`vx_std=0.35`, `wz_std=0.38`), while `velocity_smoother` owns the effective acceleration limits with `max_accel=[0.55, 0.0, 0.90]` / `max_decel=[-0.95, 0.0, -1.10]`. Ordinary Nav2/MPPI navigation allows only a bounded terminal reverse correction (`vx_min=-0.08`, `min_velocity=[-0.08, 0.0, -1.00]`) so the controller can recover a measured near-goal forward overshoot instead of stalling outside the 6 cm goal checker tolerance. `PreferForwardCritic` (`cost_weight=20.0`, `threshold_to_consider=0.50`) keeps long-range path following forward-biased and reserves reverse sampling for near-goal correction, while `PathFollowCritic.cost_weight=7.0` keeps long goals from falling into a low-speed terminal-correction local optimum. RotationShim engages above `0.45 rad` and disengages below `0.075 rad`; this reserves pure-yaw shim behavior for larger heading changes and keeps that shim active until residual heading is small. `robot_safety` checks actual `/wheel/odom` yaw-rate settle before the next linear drive command after a pure spin. `GoalCritic` enters at `1.5 m` with `cost_weight=16.0`, while `GoalAngleCritic` remains `6.0`; this biases Ackermann approach geometry toward final XY before terminal heading closure. API-owned ordinary navigation publishes a distance-to-goal `/speed_limit` because 1.2 m/s chassis tests measured about 0.58 m of stop distance; it steps down from 1.20 m/s outside 2.0 m to 0.25 m/s in the 0.15..0.6 m band, only using 0.10 m/s inside 0.15 m, and restores the 1.20 m/s cruise limit when the Nav2 task exits instead of publishing a `0.0` stop limit. If MPPI stalls within 0.30 m for 1.5 s after the 3 s minimum wait, API cancels only the active Nav2 goal and moves into bounded final verification/yaw/retry instead of letting BT-level retries spend minutes around the target. `TwirlingCritic.cost_weight=1.0` uses Humble's live parameter name and avoids the plugin default suppressing large initial heading corrections. The local planning window is `10m x 10m`, `ObstaclesCritic` is matched to `local_inflation_layer` (`inflation_radius=0.35`, `cost_scaling_factor=6.0`), and `PathAlignCritic.max_path_occupancy_ratio=0.05` so local dynamic obstacle avoidance can deviate from the global path earlier. Single-pose navigation uses the repository behavior tree directly as `ComputePathToPose -> FollowPath` after the Savitzky-Golay BT smoother was rolled back; that rollback removed the mid-route pure-spin/replan chain observed in the delivery-point loop. `behavior_server` and `smoother_server` remain part of the point-navigation lifecycle core for compatibility and tooling, but the active field BT does not call `Wait` or `SmoothPath`. `controller_server.failure_tolerance=3.0` only handles very short MPPI no-trajectory intervals; longer blocked corridors should be handled by higher-level wait/retry policy instead of hidden BT smoothing/recovery.

The BT inner local-costmap recovery and outer global-costmap recovery both use a 1 second `Wait`; no repository navigation tree intentionally parks for a 5 second recovery wait before retrying.

The legacy `robot_local_perception` balanced clearing profile is `clearing.virtual_rays.angular_resolution_deg=0.75`, `range_steps=[0.50, 1.00, 2.00, 3.50, 5.50, 8.00]`, 8 z endpoints, and `clearing.max_points=30000`. In accel profiles, the same clearing role is owned by `PointCloudAccelCore` through `clearing_worker_virtual_rays_enabled=true`, `clearing_worker_virtual_ray_angle_resolution_deg=1.0`, the same angular observation window as obstacle marking, near-body range steps starting at `0.10m`, 9 z endpoints, `clearing_worker_virtual_rays_allow_self_mask_endpoints=true`, and `clearing_worker_max_points=30000`. Obstacle marking still excludes the self mask; clearing is allowed to pass through the near-body/self-mask region so stale VoxelLayer cells cannot form a ring around the robot footprint. This keeps no-return clearing active over the local window without changing `/lidar_points`, TF, stamps, DDS/QoS, or Nav2 controller settings. `collision_monitor` uses shorter timeouts (`source_timeout=0.6`, `stop_pub_timeout=0.3`) so stale perception affects velocity commands for less time.
